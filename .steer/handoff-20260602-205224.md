Implement core TLS client certificate loading for the poo-pi Pi package, including PFX/P12 setup, hidden passphrase input, redacted provider state, lazy consumer wiring, smoke scripts, and clearer status behavior. The latest focus was UI/status clarity after `/reload` showed confusing status text.

- TLS was refactored from `extensions/core/extensions/tls.ts` into `extensions/core/extensions/tls/{index.ts,types.ts,pfx-source.ts,tui.ts,persistence.ts}`.
- `extensions/core/index.ts`, `extensions/core/extensions/proxy/index.ts`, and `extensions/core/extensions/websearch.ts` import TLS from `tls/index.ts`.

- `extensions/poo-pi.ts` no longer sets `poo-pi loaded` status.
- `extensions/core/extensions/tls/index.ts` now treats no-UI setup-required as `unconfigured`, not `error`, and notifies with `run /tls-setup interactively`.
- Validation recently passed: core import smoke, `npm run smoke:tls-consumers`, `npm run validate:json`; earlier pack dry run and fixture smoke also passed.

## What went wrong
The extension behaved as implemented but the status UX was misleading: normal “not configured yet” showed as `tls: error`, and a package-level `poo-pi loaded` status added visual noise. The latest fixes were made quickly and may need an actual Pi `/reload` interactive check to verify the footer/status layout now looks right.

## Dead ends — do not retry these
- Do not re-add `poo-pi loaded` as a persistent status; it provides no actionable state and clutters the footer.
- Do not represent “no saved cert / setup required” as `tls: error`; it is a normal unconfigured state.
- Do not expose cert path, PFX bytes, passphrase, passphrase length, decrypted cert info, or fingerprint in tool/command/status output.

## Open unknowns
- Whether `ctx.ui.notify()` in no-UI contexts is useful or visible; the current code calls it for unconfigured/error in `session_start` when `!ctx.hasUI`.
- Whether the status is actually “below the status bar” due to Pi footer rendering, a stale `/reload`, or the extension setting multiple statuses.
- No actual interactive `/tls-setup` run was performed after the status cleanup.

## Suggested next approach
Run Pi interactively with the extension and reload once to inspect the status footer. If `tls: unconfigured` is still noisy, consider clearing the TLS status entirely when unconfigured and only showing `tls: loaded` or `tls: error`. Keep `/tls-setup` as the discoverable/actionable path rather than relying on persistent footer text.
